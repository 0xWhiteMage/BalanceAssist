# Temporary Session Retention

Temporary same-browser working drafts expire 24 hours after meaningful authenticated activity. The GitHub Actions expiry worker runs every five minutes on a best-effort basis. It deletes expired sessions and their unclaimed outbox rows, but defers deletion while a dispatcher holds a valid handoff lease. Producer transfer happens only after explicit consent. Claiming atomically checks expiry and that consent; an expired or revoked unclaimed handoff is suppressed without sending and emits a non-PII `handoff_suppressed` event.

Each claim has an ownership token. Before calling Telegram, the dispatcher changes its owned `claiming` row to `sending` and reserves it for 90 seconds. Telegram calls, including response parsing, are bounded to 45 seconds. A process pause beyond the reservation, crash, or lost acknowledgement can leave acceptance unknowable; retries are at-least-once and can duplicate an externally accepted Telegram delivery. Expiry and consent revocation never retract an already accepted external transfer.

Private attachment objects are removed before their metadata changes to `expired`; failed deletion leaves metadata stored for retry. Files are analysis-only, use opaque keys, and are never transferred to Telegram or the Balance team. Migration `030` requires a private bucket, removes permissive API policies, revokes browser-role object privileges, and records readiness. Uploads remain unavailable until readiness is `ready`. Cleanup is best effort, so removal is not guaranteed while the worker or Storage is unavailable.

An authenticated deletion request creates a status-tracked job and is processed within 24 hours. `requested`, `claimed`, `processing`, and `failed` do not mean deletion completed; only `completed` confirms application-data deletion. The service cannot immediately erase provider backups or content already transferred to Telegram, which have separate retention controls.
